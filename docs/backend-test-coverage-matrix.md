Operator coverage includes `UserRole`, DB-current-role guard context, `GET /api/v1/operator/me`, admin-only user list/get, admin-only audited role change, admin-only status/restore, role/status validation, and `OperatorAuditLog` metadata redaction tests. Scheduler/Ops coverage includes disabled-by-default config, `SCHEDULER_TICK_INTERVAL_MS=60000`, dry-run-by-default scheduler calls, `OpsJobRun` audit/redaction, `OpsJobLock`, opt-in real DB concurrent lock smoke, internal runner duplicate-lock behavior, placeholder skipped/`NOT_IMPLEMENTED` assertions, scheduler flag selection, and `/readiness` mock e2e. The fixed KIS stock watchlist validates to 40 symbols within the 41-symbol limit. Local DB mapping counts passed at domestic 15/15, US 25/25, and separate Binance crypto 2/2. KIS endpoint/policy env is present after non-secret `.env.local` update, KIS approval/connect/subscribe ack evidence succeeded, domestic `H0STCNT0` tick parsing and 12 domestic `kis_krx_realtime_trade` provider_api DB inserts succeeded. A 2026-06-03 DB-started market-window US-only rerun completed with dry-run `wouldCreate=35` and non-dry-run 25 `kis_us_delayed_trade` provider_api USD DB inserts for active US stock assets. ExchangeRate/Binance dry-run rechecks succeeded. Provider_api source eligibility is now open only for `/fx quote`, `/fx execute`, assets `withPrice`, orders quote, orders execute, live portfolio/home/positions valuation, and operator-run daily snapshot valuation. Orders create binds durable quotes but does not read provider rows directly.

This document records the current backend test surface before starting the next large backend gates such as Reward Policy / Reward Catalog, provider_api final/ranking/reward eligibility, WebSocket ingestion, cron scheduling, ranking overwrite/regeneration automation, settlement extensions beyond final tier assignment, external reward/payment/point/delivery fulfillment, or further auth hardening.

This document is a coverage matrix. It does not authorize provider_api source eligibility beyond the explicit allowlist (`/fx quote`, `/fx execute`, assets `withPrice`, orders quote, orders execute, live valuation, and operator-run daily snapshot valuation), scheduler, settlement extensions, external reward/payment/point/delivery fulfillment, seed, package, or lockfile changes. Auth refresh-session, Durable Quote, admin status/restore, and internal reward fulfillment schema/migrations are now implemented; further schema changes still require a separate gate.

## Coverage Legend

- Covered: Assertions exist for the current MVP behavior.
- Partially covered: Some meaningful assertions exist, but an important runtime path, HTTP path, DB path, or race path is missing.
- Not covered: No meaningful current test was found for that test type and domain.
- Blocking before MVP launch: Whether the missing test must block the current access-token/manual-data/full-fill MVP. Future provider/scheduler/settlement/reward gates may add stronger blocking criteria.

## Coverage Conclusions

- The strongest current safety coverage is in `FX execute` and `Orders execute`, with `Season join` now covered by an opt-in PostgreSQL integration spec for atomicity, duplicate/race behavior, inactive rejection, and rollback injection.
- The strongest auth coverage is at the guard/service level plus mock e2e baseline. Refresh rotation/logout/logout-all have service and mock e2e coverage, and the opt-in Auth DB smoke covers hash-only storage and rotation/revoke behavior. Protected write paths still include valid-token service-entry smoke without provider data.
- Operator/admin coverage proves role separation, DB-current-role authorization, admin-only user list/get, admin-only role change, admin-only status/restore, refresh-session revocation on suspend/delete, success/failure audit metadata, and redaction. It does not implement batch execution, provider ingestion trigger, external reward provider trigger, or scheduler API.
- Scheduler/Ops foundation coverage proves disabled-by-default config, `SCHEDULER_TICK_INTERVAL_MS=60000`, dry-run-by-default scheduler calls, lock acquire/active/expired/release behavior, opt-in real DB same-key concurrent lock smoke, run audit success/failure/locked state, metadata/result redaction, internal daily snapshot runner lock integration, skipped/not-implemented placeholder jobs, and `/readiness` without external provider calls. It does not prove production cron ownership or real automatic writes.
- A provider-key-free `MVP_FLOW_DB_SMOKE=1` real PostgreSQL smoke now verifies that implemented services connect in one isolated MVP user flow using test-only `admin_manual` fixture rows and cleanup. It is service-composed rather than real HTTP because the existing real DB runner pattern uses `tsx` service composition.
- Batch foundation now has service/parser coverage for `BatchJobRun` lifecycle, dryRun, result/error persistence, idempotency policy, the operator-run provider-backed daily portfolio snapshot job with sourceSummary/fallback coverage, the operator-run season ranking job, the operator-run daily season cycle orchestration job, the operator-run season settlement MVP job, the operator-run final tier assignment MVP job, and the operator-run reward grant internal foundation MVP job. Cron scheduling, provider ingestion jobs, ranking overwrite/regeneration automation, settlement extensions beyond final tier assignment, and external reward/payment fulfillment remain unimplemented.
- Provider live smoke evidence now proves ExchangeRate-API and Binance public REST can insert local `provider_api` rows when opt-in env is enabled and active asset mappings exist. KIS endpoint completion now proves approval_key/connect/subscribe ack, domestic `H0STCNT0` tick parsing, domestic KRX provider_api row insertion, US `HDFSCNT0` tick parsing, and US provider_api row insertion.
- Read-only APIs have solid service-level no-mutation assertions, but only limited real DB HTTP e2e.
- Provider_api final/ranking/reward eligibility, Binance WebSocket ingestion, KIS reconnect hardening, cron scheduler, cron-driven ranking automation, settlement extensions, reward policy/catalog, actual reward/payment/point/delivery/external fulfillment, access-token blacklist, and cookie/session auth have no implementation coverage because those gates remain closed.
- Manual CLI, operator-run batch, and automatic scheduler must not be treated as equivalent: daily snapshots and rankings currently have manual/operator surfaces and unit tests, not cron-driven automatic execution.
