# Backend Test Coverage Matrix

Status: documentation/code-audit matrix, updated 2026-05-27 for KIS WebSocket trade price ingestion foundation coverage, provider ingestion foundation coverage, operator-run daily portfolio snapshot, season ranking, daily season cycle, season settlement MVP, final tier assignment MVP, reward grant internal foundation MVP batch job coverage, rewards/badges API coverage, and Home settled final-result read model coverage.

This document records the current backend test surface before starting the next large backend gates such as provider_api source eligibility, WebSocket ingestion, cron scheduling, ranking overwrite/regeneration automation, settlement extensions beyond final tier assignment, actual reward fulfillment, or further auth hardening.

This document is a coverage matrix. It does not authorize provider_api source eligibility, scheduler, settlement extensions, actual reward fulfillment, seed, package, or lockfile changes. Auth refresh-session schema/migration is now implemented; further schema changes still require a separate gate.

Crypto policy update:

- MVP crypto provider is Binance.
- Crypto is USD-settled and uses the USD Wallet.
- Crypto KRW valuation is USD crypto value converted by USD/KRW.
- Upbit/Bithumb and KRW crypto trading are excluded from MVP.
- Provider ingestion foundation exists for explicit operator-run ExchangeRate-API USD/KRW, Binance public REST crypto price snapshot insertion, and KIS WebSocket domestic/US trade price snapshot insertion.
- Provider_api source eligibility for quote, execute, valuation, daily snapshot, ranking, and settlement remains a separate gate.

## Coverage Legend

- Covered: Assertions exist for the current MVP behavior.
- Partially covered: Some meaningful assertions exist, but an important runtime path, HTTP path, DB path, or race path is missing.
- Not covered: No meaningful current test was found for that test type and domain.
- Blocking before MVP launch: Whether the missing test must block the current access-token/manual-data/full-fill MVP. Future provider/scheduler/settlement/reward gates may add stronger blocking criteria.

## Coverage Conclusions

- The strongest current safety coverage is in `FX execute` and `Orders execute`, with `Season join` now covered by an opt-in PostgreSQL integration spec for atomicity, duplicate/race behavior, inactive rejection, and rollback injection.
- The strongest auth coverage is at the guard/service level plus mock e2e baseline. Refresh rotation/logout/logout-all have service and mock e2e coverage, and the opt-in Auth DB smoke covers hash-only storage and rotation/revoke behavior. Protected write paths still include valid-token service-entry smoke without provider data.
- A provider-key-free `MVP_FLOW_DB_SMOKE=1` real PostgreSQL smoke now verifies that implemented services connect in one isolated MVP user flow using test-only `admin_manual` fixture rows and cleanup. It is service-composed rather than real HTTP because the existing real DB runner pattern uses `tsx` service composition.
- Batch foundation now has service/parser coverage for `BatchJobRun` lifecycle, dryRun, result/error persistence, idempotency policy, the operator-run daily portfolio snapshot job, the operator-run season ranking job, the operator-run daily season cycle orchestration job, the operator-run season settlement MVP job, the operator-run final tier assignment MVP job, and the operator-run reward grant internal foundation MVP job. Cron scheduling, provider jobs, ranking overwrite/regeneration automation, settlement extensions beyond final tier assignment, and actual external fulfillment remain unimplemented.
- Read-only APIs have solid service-level no-mutation assertions, but only limited real DB HTTP e2e.
- Provider_api source eligibility, Binance WebSocket ingestion, KIS live smoke/reconnect hardening, cron scheduler, cron-driven ranking automation, settlement extensions, actual reward/payment/point/delivery/external fulfillment, access-token blacklist, and cookie/session auth have no implementation coverage because those gates remain closed.
- Manual CLI, operator-run batch, and automatic scheduler must not be treated as equivalent: daily snapshots and rankings currently have manual/operator surfaces and unit tests, not cron-driven automatic execution.
